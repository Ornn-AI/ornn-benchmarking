# Operational Guidance

Learnings from validation runs and development that future workers should know.

---

## Service Restart After Auth-Key Config Changes

When modifying API key configuration (adding, revoking, or changing keys in `api/config.py`), the running API process **must be restarted** before those changes take effect. The API server reads its configuration at startup and caches it in memory.

**Symptoms of stale config:**
- Newly added keys are rejected with 401
- Revoked keys still accepted
- Rate-limit settings don't apply

**Fix:** Stop and restart the API service using the manifest commands:
```bash
# Stop
pids=$(lsof -ti :8080); [ -z "$pids" ] || kill $pids

# Start
FIRESTORE_EMULATOR_HOST=127.0.0.1:8085 PORT=8080 ./.venv/bin/python -m uvicorn api.main:app --host 0.0.0.0 --port 8080
```

## Firestore Emulator Java Prerequisite

The Firestore emulator (`gcloud emulators firestore start`) requires a Java runtime (JDK 11+). On machines without Java:
- Use the in-memory fake Firestore backend for local development and testing
- Tests use `conftest.py` fixtures that provide a protocol-compatible fake client
- CI/CD does not require Java — tests run with mocked Firestore

## CI Workflow Design

The GitHub Actions CI workflow (`.github/workflows/ci.yml`) is designed to be:
- **Free for public repos** — uses only GitHub-hosted ubuntu-latest runners
- **No secrets required** — all jobs run without credentials
- **No external services** — no database, emulator, or API dependencies
- **Multi-Python testing** — tests run against Python 3.10 and 3.12

## Rate Limiting in Tests

The API rate limiter is configured per-key. When running parallel test suites:
- Each test client should use a distinct API key
- The default test limit is 3 requests per 10 seconds per key
- Tests that exceed this will receive 429 responses
- Wait 10 seconds between bursts of 3 requests with the same key

## Report JSON Files at Repo Root

Files matching `ornn_report_*.json` at the repo root are runtime-generated benchmark reports. They are covered by `.gitignore` and should never be committed. If they appear in `git status`, they were likely generated by a test or manual run.
